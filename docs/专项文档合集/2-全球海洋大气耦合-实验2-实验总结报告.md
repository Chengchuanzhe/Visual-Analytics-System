### （三）实验选题

**1. 自选题名称**：全球海洋-大气耦合时空可视分析系统

**2. 选题背景与意义**：

（1）实际应用场景：该系统主要服务于气候预测、台风/风暴潮预警、海洋生态保护及远洋航运规划等领域。

（2）可视化要解决的核心问题：在实验1数据整理与基础绘图的基础上，实验2进一步聚焦高级图表设计开发、视觉编码设计优化与系统前端界面原型搭建。通过桑基图揭示海域→温度区间的能量流动结构，通过平行坐标图实现跨维度耦合关系的交互式探索，解决传统单一图表无法同时表达"海域差异 + 多维耦合 + 时空演变"的痛点。

（3）展示的维度：时间（time）、经度（longitude）、纬度（latitude）、海域（region）、海表温度（sst）、盐度（salinity）、风速（wind_speed）、风向（wind_dir）、气压（pressure）、降水量（precipitation）、CO₂浓度（co2）、浪高（wave_height）、大气湿度（humidity）、海流速度（current_speed）、叶绿素浓度（chlorophyll），共 16 个维度。

**3. 数据说明**：

- **数据来源**：IFREMER Argo 浮标实测数据（海表温度、盐度） + ERA5 气候学估算（大气变量）
- **原始规模**：通过 ERDDAP REST API 按 6 个海域分别采集，原始记录总量超过百万条
- **清洗后规模**：每海域随机采样 600 条，共 3600 条结构化 JSON 数据
- **数据维度**（16维）：id、time、longitude、latitude、region（基础信息 5 维）+ sst、salinity（Argo 实测 2 维）+ wind_speed、wind_dir、pressure、precipitation、co2、wave_height、humidity、current_speed、chlorophyll（气候学估算 9 维）
- **时间跨度**：2020-01-01 至 2022-12-31（3 年）
- **地域覆盖**：太平洋、大西洋、印度洋、北极海域、南极海域、赤道区域（6 个海域）

```javascript
      .on("end", event => onBrushEnd(event, dim.key));
    // ...
  });
}

// 多重刷选 AND 逻辑
function applyAllFilters() {
  polylines.each(function(d) {
    let pass = true;
    for (const key of activeFilterKeys) {
      const [lo, hi] = brushFilters[key];
      const v = d[key];
      if (v < lo || v > hi) { pass = false; break; }
    }
    d3.select(this).classed("filtered-out", !pass);
  });
}

// 维度芯片切换（5-11 维自由增减）
chips.append("button")
  .classed("active", activeDims.includes(dim))
  .on("click", function() {
    if (activeDims.includes(dim)) {
      if (activeDims.length <= 5) { alert("至少保留 5 个维度"); return; }
      activeDims = activeDims.filter(d => d !== dim);
    } else {
      activeDims = [...activeDims, dim];
    }
    fullRender();
  });
```

## 四、实验难点与解决方法

本部分记录实验各阶段遇到的核心技术难点、问题现象与解决方法，按个人主责难点 + 小组协同难点分模块记录。

### （一）个人主责难点与解决方法

#### 1. 数据统筹与基础图表开发（主责：王哲）

**难点1：Argo 数据下载效率与内存控制**
- 问题现象：6 个海域的 Argo 原始数据总量超过百万条，单次加载会导致内存溢出；ERDDAP API 有请求速率限制，连续快速请求会被拒绝。
- 解决思路与方法：引入分海域分批次请求策略，每次请求间隔 1 秒（礼貌性限速）；每海域仅采样保留 600 条有效记录；使用 Python `requests` 库的 `timeout=180` 超时控制；过滤掉关键字段（time/latitude/longitude/temp）为 null 的记录。
- 最终效果：成功获取 3600 条有效记录，数据下载过程稳定无中断。

**难点2：大气变量估算的物理合理性**
- 问题现象：9 个大气变量（风速、气压、降水、CO₂等）需要基于气候学规律估算，直接使用随机数会导致物理量级不合理（如赤道风速过大、极地降水过高）。
- 解决思路与方法：基于纬度（`math.sin(lat_r)`）和季节（`math.sin(2π × (month-1)/12)`）构建气候学模型函数 `estimate_atmo()`；使用高斯分布 `random.gauss(mean, std)` 生成合理波动；利用 AI 辅助校验各变量的纬度-季节分布合理性。
- 最终效果：估算值在物理约束范围内，满足可视化分析需求。

**难点3：多图表数据格式统一**
- 问题现象：饼图需要按海域统计的汇总格式、气泡图需要点坐标格式、雷达图需要归一化格式、桑基图需要节点/边格式，四种格式差异巨大，每个图表独立转换会导致大量重复代码。
- 解决思路与方法：设计统一数据适配模块 `data-adapter.js`，提供 `toPieData`、`toPointData`、`toRadarData`、`toSankeyData` 四个格式转换函数；使用闭包缓存已加载的数据避免重复请求；暴露 `COLORS` 和 `REGIONS` 常量供各图表统一引用。
- 最终效果：后续所有图表（B/C/D）均可通过 `Adapter.load().then(data => ...)` 一行代码加载数据，零重复代码。

---

#### 2. 高级图表与基础图表开发（主责：许一凡）

**难点1：桑基图节点布局与边路径计算**
- 问题现象：使用 `d3-sankey` 插件时，节点位置计算出现部分节点高度为 0 或重叠，边路径显示异常。
- 解决思路与方法：使用 `d3.sankey().nodeWidth(24).nodePadding(16)` 调整节点宽度与间距；通过 `extent([[0, 0], [IW, IH]])` 设置布局范围；区分海域节点（左侧，有颜色）和温度区间节点（右侧，灰色）的填充策略。
- 最终效果：10 个节点位置分布均匀、边流向清晰。

**难点2：3600 点散点图的重叠遮挡**
- 问题现象：3600 个点在同一坐标系中高度重叠，用户难以辨识数据分布密度与异常值。
- 解决思路与方法：使用 `opacity: 0.7` 半透明渲染，重叠区域颜色叠加增强视觉密度感；悬浮时将目标点 `r` 从 4 → 6 并提升 `opacity` 至 1，同时降低其他点透明度；使用 `z-order` 动态排序。
- 最终效果：密度分布可视化清晰，单点交互精确。

**难点3：雷达图极坐标映射与标签角度**
- 问题现象：雷达图 6 个轴的标签在某些角度下与多边形重叠，阅读困难。
- 解决思路与方法：使用标准极坐标公式计算多边形顶点 `(r·cos(θ-π/2), r·sin(θ-π/2))`；标签位置偏移至多边形外侧（`r + padding`）；动态调整 `text-anchor` 避免文字与图形重叠。
- 最终效果：6 轴标签清晰可读，6 海域轮廓区分明显。

#### 3. 新颖图表与视觉优化（主责：刘国宁）

**难点1：平行坐标图大数据量渲染性能**
- 问题现象：3600 条全量数据 × 8 维 = 28800 个点的折线渲染导致页面帧率下降至 15-20 FPS，交互响应延迟明显。
- 解决思路与方法：实现"采样模式"（max 750 条，均匀采样）作为默认视图，用户可手动切换"全量模式"；使用 `d3.scalePoint()` 精确映射轴位置；采样步长 = `Math.floor(data.length / 750)`。
- 最终效果：采样模式下 FPS 稳定 60，全量模式下 30-40 FPS，用户可根据需求灵活切换。

**难点2：多重轴刷选的 AND 逻辑实现**
- 问题现象：当用户在 3 个轴上同时拖选范围时，过滤逻辑需要取交集（AND），且刷选范围需在清除时不残留状态。
- 解决思路与方法：使用 `brushFilters` 对象管理各轴的过滤范围 `{ dimKey: [lo, hi] }`；`applyAllFilters()` 中遍历所有活跃过滤键，对每条折线执行 AND 逻辑；`onBrushEnd` 中处理 `!event.selection` 时删除对应过滤键。
- 最终效果：任意组合的轴刷选均正确取交集，重置按钮一键清除所有过滤。

**难点3：色盲友好配色的可视区分度平衡**
- 问题现象：6 个海域需要 6 种颜色，传统色板中绿色/红色在色盲人群中无法区分，而蓝色系列的邻近色对普通用户分辨度又不够。
- 解决思路与方法：采用 Okabe-Ito 色板（专为色盲友好设计的 7 色方案），选取其中 6 色；使用 Color Safe 工具模拟红绿色盲/蓝黄色盲视角验证；确保任意两色在色盲模拟下仍有足够对比度。
- 最终效果：6 海域颜色在正常视觉和色盲模拟下均可清晰区分，满足 WCAG AA 标准。

#### 4. 交互设计与前端原型搭建（主责：程传哲）

**难点1：iframe 嵌入图表的路径管理**
- 问题现象：各图表 HTML 文件分散在 `code/week4/`、`code/week5/`、`code/week7/` 等不同目录中，数据加载的相对路径（如 `../../../data/...`）在 iframe 中以图表自身目录为基准，而非原型页面目录。
- 解决思路与方法：保持各图表独立运行时的路径不变；在前端原型中通过 `MEMBER_CHARTS` 配置对象管理每个图表的精确 src 路径（如 `../../week4/模块2-图表开发/...`）；使用 `onerror` 回调处理加载失败的 iframe。
- 最终效果：所有 6 个图表通过 iframe 正确加载，数据路径无冲突。

**难点2：一次预加载 vs 按需加载的权衡**
- 问题现象：6 个图表均含 3600 条数据，全部同时加载会导致浏览器卡顿；但按需加载又会导致每次切换图表时有明显等待。
- 解决思路与方法：采用"页面初始化时一次性预加载全部 iframe"，利用浏览器并行加载能力；每个图表只加载一次，后续切换仅做 CSS display 切换（`chart-panel.active`），无额外网络请求；`bodyEl.dataset.loaded = "1"` 标记避免重复加载。
- 最终效果：首次页面加载约 3-4 秒（6 图并行），之后所有切换均为即时响应（0 延迟）。

### （二）小组协同难点与解决方法

**难点1：多图表数据格式差异大，协同转换成本高**
- 问题现象：饼图（汇总格式）、气泡图（点坐标格式）、雷达图（归一化格式）、桑基图（节点/边格式）、平行坐标图（全量格式）五种数据格式差异显著，初始阶段各成员独立实现转换逻辑，出现代码重复、格式不一致、颜色映射冲突等问题。
- 解决思路与方法：由王哲（A）主责开发统一数据适配模块 `data-adapter.js`，许一凡（B）协助确认桑基图/雷达图数据结构；该模块提供 4 个格式转换函数 + 统一颜色映射；所有后续图表（B/C/D）统一通过 `Adapter.load()` 和 `Adapter.toXxxData()` 获取数据，保证数据层一致性。
- 最终效果：消除全部重复数据转换代码，新增图表只需调用适配器即可获得正确格式的数据。统一的颜色出口（`Adapter.getColor()`）确保全系统颜色一致性。

**难点2：视觉规范执行周期长，图表视觉统一反复迭代**
- 问题现象：Week 4-5 期间各图表使用各自的临时配色方案，Week 6 刘国宁（C）制定视觉规范后，Week 8-9 才完成全图表的统一，期间存在"先开发后统一"的滞后问题，导致反复修改。
- 解决思路与方法：Week 6 起明确视觉规范（Okabe-Ito 色板 + 字体/字号/间距）；Week 8 由刘国宁（C）完成颜色编码审计并确定最终方案；Week 9 全员一次性应用统一颜色表并互相审核；建立颜色对照清单避免误用。
- 最终效果：全部图表颜色、字体、间距在 Week 9 实现统一，视觉一致性满足要求。

**难点3：四大模块代码归属边界模糊，集成冲突频发**
- 问题现象：成员 D 在进行前端原型集成时，需要理解 A/B/C 三个模块的 6 个独立 HTML 文件的内部结构；修改某个图表文件可能影响前端原型的展示效果，缺乏明确的"独立运行"与"嵌入展示"边界约定。
- 解决思路与方法：采用 iframe 隔离策略——每个图表保持为完整的独立 HTML 文件（可单独双击打开运行），前端原型通过 iframe 嵌入而非直接操作图表 DOM；各图表内部代码与原型完全解耦，互不影响；交互接口通过"独立展示页面 + 前端原型 iframe 层"两层架构划分。
- 最终效果：从未出现集成冲突；各图表独立可运行、可独立调试；集成仅需配置 src 路径。
